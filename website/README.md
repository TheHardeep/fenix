# Fenix — Developer Documentation Site

A self-contained, professional documentation website for the **Fenix** Indian-broker trading
library. The design follows the Notion Developer Docs layout (top tabs · left sidebar · right
"on this page" TOC) with a CCXT-style content structure and a phoenix-orange brand.

## Running it

The site loads its Markdown content with `fetch()`, which browsers block on `file://`, so it
must be served over `http://`. Any static server works.

**Option A — bundled launcher (opens your browser automatically):**

```bash
python serve.py            # → http://localhost:8080
python serve.py 9000       # custom port
```

**Option B — Python's built-in server:**

```bash
python -m http.server 8080
# then open http://localhost:8080
```

Then browse to the printed URL.

### Documentation sections (top tabs)

- **Guides** — overview, install, quickstart.
- **Core Concepts** — architecture, `describe()`, `_API`, maps, constants, unified JSON, logging, rate limiting, errors.
- **Unified API** — authentication, tokens, orders, positions, account, paper mode.
- **Brokers** — supported-brokers index, "adding a broker", and a reference page **per broker** (auto-generated).
- **Fenix-Pro** — the paid WebSocket market-data product: overview, install, quickstart, callbacks, subscriptions, transports, data contracts, errors, pricing, and a page **per live-feed adapter**.
- **Examples** — cookbook, changelog.

### Regenerating broker pages

The 30 per-broker reference pages (15 `broker-*.md` for `fenix`, 15 `pro-broker-*.md` for
`fenixpro`) are generated by statically inspecting the library source — no imports, so the
brokers' optional dependencies aren't required:

```bash
python tools/gen_broker_pages.py
```

It reads `D:\Work\Fenix\fenix\fenix` and `D:\Work\Fenix\fenixpro\fenixpro`, extracts each
adapter's `describe()`, `_API`, rate limits, auth params, and public method signatures, and
writes the Markdown pages. Adjust those source paths at the top of the script if the repos move.

> **Caching:** `index.html` loads the CSS/JS with a `?v=N` query string. After editing
> `styles.css`, `nav.js`, or `app.js`, bump that number (or hard-reload) so browsers fetch the
> new version. Markdown content is always fetched fresh.

External libraries (`marked`, `highlight.js`) are loaded from a CDN, so an internet connection
is needed on first load.

## Features

- **Client-side SPA** with hash routing (`#/<page>` and deep links `#/<page>/<heading>`).
- **Dark / light theme** toggle, persisted to `localStorage`.
- **Full-text search** (`Ctrl/Cmd + K`) over titles, headings, and body text.
- **Auto-generated "On this page"** TOC with scroll-spy.
- **Code blocks** with language labels, syntax highlighting, and copy buttons.
- **Callouts, cards, tables, broker chips,** and an architecture diagram.
- **Responsive** down to mobile, with a collapsible sidebar.

## Editing content

Each page is a Markdown file in `content/`. To change a page, edit its `.md` file — no build
step. To add a page, create `content/<id>.md` and register it in the appropriate group in
`assets/js/nav.js`.

Callouts use GitHub-style blockquotes:

```markdown
> [!TIP] Optional title
> Body of the callout.
```

Supported types: `TIP`, `INFO`/`NOTE`, `WARN`/`WARNING`, `DANGER`.

## Authoring source

Content is grounded in the Fenix source tree (`fenix/base/broker.py`, `constants.py`,
`errors.py`, the broker adapters, and the `fenix/paper/` package). It documents the current
`load_*_tokens` / `authenticate` / `fetch_*` API surface.
