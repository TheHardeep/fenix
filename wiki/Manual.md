# Fenix v2 Manual

## Overview

Fenix v2 is a major refactor of the library. Broker adapters are now class-based
instances, the common order helpers are generated by the shared `Broker` base
class, and broker-specific adapters such as `AliceBlue` implement the live API
details.

This manual uses `fenix.aliceblue.AliceBlue` as the reference adapter because it
shows the full v2 shape:

- `authenticate()` for live login or reusable headers.
- `load_*_tokens()` methods for instrument metadata.
- `place_order()` plus generated order helpers.
- `fetch_*()` account and order reads.
- `modify_order()`, `cancel_order()`, `square_off_position()`, and
  `exit_bracket_order()`.
- `paper_mode=True` for in-process paper trading.
- Token-bucket rate limiting inherited from `Broker`.
- Redacted logging and last request/response diagnostics.

The old v1 `create_order_eq()`, `create_order_fno()`, `create_order()`,
`create_eq_tokens()`, `create_fno_tokens()`, and standalone paper broker model
are not part of the v2 public API.

## AliceBlue Quick Start

```python
from fenix import AliceBlue, Product, Side, UniqueID, Validity

broker = AliceBlue({
    "user_id": "YOUR_USER_ID",
    "password": "YOUR_PASSWORD",
    "totpstr": "YOUR_TOTP_SECRET",
    "api_key": "YOUR_API_KEY",
    "api_secret": "YOUR_API_SECRET",
})

headers = broker.authenticate()

equity_tokens, all_tokens = broker.load_equity_tokens()
token = all_tokens["12345_NSE"]  # use a real "{Token}_{Exchange}" key

order = broker.limit_order(
    token,
    quantity=1,
    side=Side.BUY,
    price=2500.0,
    product=Product.MIS,
    validity=Validity.DAY,
    unique_id=UniqueID.LIMIT_ORDER,
)

print(order)
```

## AliceBlue Configuration

`AliceBlue(config=None)` accepts a dictionary passed to the base `Broker`.

Common live credentials:

| Key | Purpose |
| --- | --- |
| `user_id` | AliceBlue user id |
| `password` | AliceBlue password |
| `totpstr` | TOTP secret used to generate the current OTP |
| `api_key` | AliceBlue API/vendor key |
| `api_secret` | API secret used to derive the checksum |

Common runtime options:

| Key | Purpose |
| --- | --- |
| `paper_mode` | Route supported order and read methods through the paper engine |
| `paper_client_id` | Client id shown by paper profile data |
| `paper_starting_margin` | Starting available margin for paper mode |
| `paper_log_history_size` | Number of paper interactions to keep |
| `enableRateLimit` | Enable or disable the broker rate limiter |
| `rate_limit_padding` | Safety multiplier applied to computed wait time |
| `logger` | Python logger used for debug output |
| `verbose` | Enable verbose request/response logging |
| `logSensitive` | Allow sensitive values in logs when explicitly set to `True` |
| `maxLogBodyLength` | Truncate long logged bodies to this length |

## Authentication

Signature:

```python
broker.authenticate(
    params: dict | None = None,
    headers: dict[str, str] | None = None,
    force: bool = False,
) -> dict[str, str]
```

Use credentials from the broker config:

```python
broker = AliceBlue({
    "user_id": "YOUR_USER_ID",
    "password": "YOUR_PASSWORD",
    "totpstr": "YOUR_TOTP_SECRET",
    "api_key": "YOUR_API_KEY",
    "api_secret": "YOUR_API_SECRET",
})

headers = broker.authenticate()
```

Pass credentials directly:

```python
headers = broker.authenticate(params={
    "user_id": "YOUR_USER_ID",
    "password": "YOUR_PASSWORD",
    "totpstr": "YOUR_TOTP_SECRET",
    "api_key": "YOUR_API_KEY",
    "api_secret": "YOUR_API_SECRET",
})
```

Reuse stored headers:

```python
headers = broker.authenticate(headers={
    "ID": "YOUR_USER_ID",
    "AccessToken": "ACCESS_TOKEN",
    "Authorization": "Bearer YOUR_USER_ID ACCESS_TOKEN",
    "X-SAS-Version": "2.0",
    "User-Agent": "AliceBlue_V21.0.1",
    "Content-Type": "application/json",
    "susertoken": "SUSER_TOKEN",
})
```

Use `force=True` to ignore cached headers and repeat the login flow.

In paper mode, `authenticate()` returns `{"paper": "true"}` and does not call
AliceBlue.

## Instrument Tokens

AliceBlue v2 token methods are named `load_*_tokens()`. They return a tuple:

```python
segment_map, all_token_lookup = broker.load_equity_tokens()
```

The second value is keyed by `"{Token}_{Exchange}"` and is usually the easiest
way to retrieve a `token_dict` for order placement.

### Equity Tokens

```python
equity, all_tokens = broker.load_equity_tokens()
```

Signature:

```python
load_equity_tokens(data: dict[str, list] | None = None) -> tuple[dict, dict]
```

Expected returned structure:

```python
{
    "Equity": {
        "NSE": {"RELIANCE": {...}},
        "BSE": {"RELIANCE": {...}},
    }
}
```

Each record contains:

- `Token`
- `Exchange`
- `Symbol`
- `ScriptName`
- `LotSize`
- `TickSize`

### Index Tokens

```python
indices, all_tokens = broker.load_index_tokens()
```

Signature:

```python
load_index_tokens(data: dict[str, list] | None = None) -> tuple[dict, dict]
```

Returned map:

```python
{
    "Indices": {
        "NSE": {...},
        "BSE": {...},
        "MCX": {...},
    }
}
```

### Futures and Options Tokens

```python
fno, all_tokens = broker.load_fno_tokens()
```

Signature:

```python
load_fno_tokens(data: dict[str, list] | None = None) -> tuple[dict, dict]
```

Returned map:

```python
{
    "Futures": {"NFO": {...}, "BFO": {...}},
    "Options": {"NFO": {...}, "BFO": {...}},
}
```

Derivative records include fields such as:

- `Exchange`
- `Token`
- `Root`
- `Symbol`
- `TickSize`
- `LotSize`
- `Expiry`
- `StrikePrice` for options
- `Option` for options
- `ScriptName`

### MCX Tokens

Returned map:

```python
    "Futures": {"MCX": {...}},
```

## Token Dictionary For Orders

All order methods require a `token_dict` with at least:

```python
token = {
    "Token": 12345,
    "Exchange": "NSE",
    "Symbol": "RELIANCE-EQ",
}
```

The easiest path is to use the `all_tokens` lookup returned by a token loader:

```python
_, all_tokens = broker.load_equity_tokens()
token = all_tokens["12345_NSE"]
```

## Order Placement

AliceBlue implements `place_order()` directly. The base `Broker` class generates
convenience helpers that call `place_order()` with fixed order-type defaults.

### `place_order()`

Signature:

```python
broker.place_order(
    token_dict: dict,
    quantity: int,
    side: str,
    product: str,
    validity: str,
    variety: str,
    unique_id: str,
    price: float = 0.0,
    trigger: float = 0.0,
    target: float = 0.0,
    stoploss: float = 0.0,
    trailing_sl: float = 0.0,
) -> dict
```

Example:

```python
from fenix import OrderType, Product, Side, Validity, Variety

order = broker.place_order(
    token_dict=token,
    quantity=1,
    side=Side.BUY,
    product=Product.MIS,
    validity=Validity.DAY,
    variety=Variety.REGULAR,
    unique_id="DirectOrder1",
    price=2500.0,
    trigger=0.0,
)
```

AliceBlue resolves order type from `price` and `trigger`:

| Price | Trigger | Resolved order type |
| --- | --- | --- |
| `0` | `0` | `MARKET` |
| `>0` | `0` | `LIMIT` |
| `>0` | `>0` | `SL` |
| `0` | `>0` | `SLM` |

### Generated Convenience Helpers

These methods are generated on `Broker` and inherited by AliceBlue.

Side-agnostic helpers:

```python
broker.market_order(token, quantity=1, side=Side.BUY, unique_id="M1")
broker.limit_order(token, quantity=1, side=Side.BUY, price=2500.0, unique_id="L1")
broker.sl_order(token, quantity=1, side=Side.BUY, price=2501.0, trigger=2500.0, unique_id="SL1")
broker.slm_order(token, quantity=1, side=Side.BUY, trigger=2500.0, unique_id="SLM1")
```

Side-specific helpers:

```python
broker.market_buy_order(token, quantity=1, unique_id="MB1")
broker.market_sell_order(token, quantity=1, unique_id="MS1")
broker.limit_buy_order(token, quantity=1, price=2500.0, unique_id="LB1")
broker.limit_sell_order(token, quantity=1, price=2500.0, unique_id="LS1")
broker.sl_buy_order(token, quantity=1, price=2501.0, trigger=2500.0, unique_id="SLB1")
broker.sl_sell_order(token, quantity=1, price=2499.0, trigger=2500.0, unique_id="SLS1")
broker.slm_buy_order(token, quantity=1, trigger=2500.0, unique_id="SLMB1")
broker.slm_sell_order(token, quantity=1, trigger=2500.0, unique_id="SLMS1")
```

Optional order fields accepted by generated helpers:

- `target`
- `stoploss`
- `trailing_sl`
- `product`
- `validity`
- `variety`

### Bracket Orders

AliceBlue sends a bracket order when `target` is non-zero. The payload then
includes:

- `targetLegPrice`
- `slLegPrice`
- `trailingSlAmount`
- bracket-order variety

```python
order = broker.limit_buy_order(
    token,
    quantity=1,
    price=2500.0,
    target=2525.0,
    stoploss=2485.0,
    trailing_sl=2.0,
    unique_id="Bracket1",
)
```

### Raw Order History


Returns raw AliceBlue order-history rows.

### Unified Order Book


Returns a list of unified Fenix order records.

### Unified Trade Book

```python

```

Returns a list of unified Fenix order/trade records.

### Single Order

```python
order = broker.fetch_order(order_id="ORDER_ID")
```

book.

### Order History

```python
history = broker.fetch_order_history(order_id="ORDER_ID")
```

Returns unified order-history records for the requested order.

## Positions, Holdings, Margin, and Profile

AliceBlue exposes these account methods:

```python
day_positions = broker.fetch_day_positions()
net_positions = broker.fetch_net_positions()
holdings = broker.fetch_holdings()
margins = broker.fetch_margin_limits()
profile = broker.fetch_profile()
```

Notes:

- `fetch_day_positions()` returns AliceBlue day-position rows in live mode.
- `fetch_net_positions()` returns AliceBlue net-position rows in live mode.
- `fetch_holdings()` returns AliceBlue holding rows in live mode.
- `fetch_margin_limits()` returns a unified Fenix RMS record.
- `fetch_profile()` returns a unified Fenix profile record.
- In paper mode, these methods return paper engine account state.

## Paper Trading

Paper trading is enabled per broker instance:

```python
from fenix import AliceBlue, Side

broker = AliceBlue({
    "paper_mode": True,
    "paper_client_id": "PAPER001",
    "paper_starting_margin": 1000000,
})

broker.authenticate()

token = {
    "Token": 12345,
    "Exchange": "NSE",
    "Symbol": "TESTSTOCK",
}

order = broker.market_buy_order(
    token,
    quantity=10,
    unique_id="PaperMarketBuy",
)

broker.on_tick(token=12345, ltp=100.0)

print(broker.fetch_orderbook())
print(broker.fetch_tradebook())
print(broker.fetch_net_positions())
print(broker.fetch_margin_limits())
```

`on_tick()` is defined on the base broker:

```python
broker.on_tick(
    token=12345,
    ltp=100.0,
    bid=99.95,
    ask=100.05,
)
```

Paper mode supports:

- Market, limit, SL, and SLM matching.
- Order modification and cancellation.
- Square-off operations.
- Order book, trade book, order history, positions, holdings, profile, and
  margin reads.
- Debug snapshots through `last_paper_request`, `last_paper_response`, and
  `last_paper_interaction`.

## Rate Limits

Fenix v2 initializes rate-limit buckets in the shared `Broker` base class. Each
broker adapter can define a `rateLimits` map in `describe()`. AliceBlue defines a
default group equivalent to 1800 requests per 15 minutes.

The fetch layer calls:

```python
broker.throttle(endpoint_group="default")
```

before sending a request. The throttler:

- Finds all token buckets for the endpoint group.
- Refills buckets according to elapsed time.
- Sleeps when a bucket does not have enough tokens.
- Applies `rate_limit_padding` to the computed wait time.
- Uses locks so concurrent calls do not over-consume the same bucket.

Default AliceBlue-style configuration:

```python
broker = AliceBlue({
    "enableRateLimit": True,
    "rate_limit_padding": 1.1,
})
```

Disable throttling only when another layer controls request pacing:

```python
broker = AliceBlue({
    "enableRateLimit": False,
})
```

Custom broker adapters can define buckets like this:

```python
{
    "rateLimits": {
        "default": {
            "period": 900,
            "capacity": 1800,
            "cost": 1.0,
        }
    }
}
```

Use the same endpoint group name in `fetch(..., endpoint_group="default")`.

## Logging and Diagnostics

Every broker instance keeps the latest HTTP request context:

- `last_request_method`
- `last_request_url`
- `last_request_headers`
- `last_request_body`
- `last_request_params`
- `last_response_headers`
- `last_response_body`
- `last_http_response`
- `last_json_response`

Enable debug logging:

```python
import logging
from fenix import AliceBlue

logger = logging.getLogger("fenix")
logger.setLevel(logging.DEBUG)

broker = AliceBlue({
    "paper_mode": True,
    "logger": logger,
    "verbose": True,
})
```

Sensitive values such as passwords, tokens, API keys, authorization headers,
TOTP values, and user profile fields are redacted in logs.

Useful logging options:

| Option | Purpose |
| --- | --- |
| `logger` | Python logger used by broker debug logs |
| `verbose` | Also emit CCXT-style verbose logs through `broker.log()` |
| `sensitiveLogKeys` | Additional keys to redact |
| `sensitiveLogKeysIncludeDefault` | Keep the built-in sensitive key set enabled |
| `logSensitive` | Disable redaction when explicitly set to `True` |
| `maxLogBodyLength` | Truncate logged payloads |
| `enableLastHttpResponse` | Store the last raw response object |
| `enableLastJsonResponse` | Store the last parsed JSON response |
| `enableLastResponseHeaders` | Store last response headers |

Paper-mode operations are logged through the same redaction path and expose:

- `last_paper_request`
- `last_paper_response`
- `last_paper_interaction`

Paper interactions also update the generic last-request fields with a synthetic
URL such as `paper://AliceBlue/place_order`, which makes paper and live
diagnostics easier to inspect with the same tooling.

## Removed or Deprecated v1 APIs

The v2 refactor removed the old segment-specific order creators. Do not use:

- `create_order_eq()`
- `create_order_fno()`
- `create_order()`
- `create_bo_order()`
- `create_co_order()`
- `market_order_eq()`
- `market_order_fno()`
- `limit_order_eq()`
- `limit_order_fno()`
- `sl_order_eq()`
- `sl_order_fno()`
- `slm_order_eq()`
- `slm_order_fno()`
- `create_eq_tokens()`
- `create_fno_tokens()`
- standalone `paper` broker imports

Use the v2 replacements:

| v1 style | v2 replacement |
| --- | --- |
| `create_eq_tokens()` | `load_equity_tokens()` |
| `create_fno_tokens()` | `load_fno_tokens()` |
| `create_order_eq()` / `create_order_fno()` | `place_order()` or generated helpers |
| `market_order_eq()` / `market_order_fno()` | `market_order()` or side-specific helpers |
| `limit_order_eq()` / `limit_order_fno()` | `limit_order()` or side-specific helpers |
| `sl_order_eq()` / `sl_order_fno()` | `sl_order()` or side-specific helpers |
| `slm_order_eq()` / `slm_order_fno()` | `slm_order()` or side-specific helpers |
| standalone paper broker | `BrokerClass({"paper_mode": True})` |

## Migration Checklist

- Instantiate broker classes, for example `broker = AliceBlue(config)`.
- Replace old `create_*_tokens()` calls with `load_*_tokens()`.
- Replace old segment-specific order creators with `place_order()` or generated
  helpers.
- Pass a real `token_dict` from a token loader to every order method.
- Use `paper_mode=True` on a real broker class for dry runs.
- Review rate-limit settings if you previously throttled calls outside Fenix.
- Configure a logger during migration so request/response redaction and
  diagnostic snapshots can help identify broker-specific API changes.
- Update code that imported removed broker modules: `choice`, `kotak`,
  `kunjee`, or `vpc`.
- Retest order flows broker by broker because v2 normalizes behavior through the
  shared base class but each broker still has API-specific limits.
